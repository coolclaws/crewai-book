# 第 18 章　A2A：Agent-to-Agent 通信协议

在多 Agent 系统中，不同 Agent 之间的通信是核心挑战之一。CrewAI 对 Google 提出的 A2A（Agent-to-Agent）协议进行了深度集成，使得 CrewAI Agent 不仅可以作为 client 委托任务给远程 Agent，还能作为 server 对外暴露 A2A 接口。本章将深入分析 `crewai/a2a/` 目录下的完整实现。

## 18.1 A2A 协议概述

A2A 是 Google 在 2024 年提出的 Agent 间通信开放标准。其核心思想是：每个 Agent 作为一个独立服务，通过标准化的 AgentCard 描述自身能力，通过 JSON-RPC、gRPC 或 HTTP+JSON 等传输协议交换任务消息。

CrewAI 在 `a2a/types.py` 中定义了支持的传输协议和协议版本：

```python
TransportType = Literal["JSONRPC", "GRPC", "HTTP+JSON"]
ProtocolVersion = Literal[
    "0.2.0", "0.2.1", "0.2.2", "0.2.3",
    "0.2.4", "0.2.5", "0.2.6",
    "0.3.0", "0.4.0",
]
```

协议版本从 0.2.0 到 0.4.0，反映了 A2A 标准的快速演进。CrewAI 通过 `ProtocolVersion` 类型约束确保只使用已知版本，默认版本为 `"0.3.0"`。

A2A 通信的核心数据流如下：

```
CrewAI Agent (Client)                   Remote Agent (Server)
     |                                        |
     |--- Fetch AgentCard ------------------->|
     |<-- AgentCard (capabilities, skills) ---|
     |                                        |
     |--- Send Task (JSON-RPC/gRPC) -------->|
     |<-- TaskState (completed/input_required)|
     |                                        |
     |--- Follow-up Message (multiturn) ----->|
     |<-- Final Result -----------------------|
```

## 18.2 配置体系：A2AClientConfig 与 A2AServerConfig

CrewAI 将 A2A 配置拆分为 client 和 server 两个独立配置类，定义在 `a2a/config.py` 中。

### 18.2.1 A2AClientConfig

`A2AClientConfig` 控制连接到远程 A2A Agent 的所有参数：

```python
class A2AClientConfig(BaseModel):
    endpoint: Url = Field(description="A2A agent endpoint URL")
    auth: ClientAuthScheme | None = Field(default=None)
    timeout: int = Field(default=120, description="Request timeout in seconds")
    max_turns: int = Field(default=10)
    response_model: type[BaseModel] | None = Field(default=None)
    fail_fast: bool = Field(default=True)
    trust_remote_completion_status: bool = Field(default=False)
    updates: UpdateConfig = Field(default_factory=_get_default_update_config)
    accepted_output_modes: list[str] = Field(
        default_factory=lambda: ["application/json"]
    )
    extensions: list[str] = Field(default_factory=list)
    client_extensions: list[ValidatedA2AExtension] = Field(default_factory=list)
    transport: ClientTransportConfig = Field(default_factory=ClientTransportConfig)
```

其中几个关键字段值得深入理解：

- **`max_turns`**：控制多轮对话的最大轮次，默认 10 轮，防止 Agent 陷入无限对话循环
- **`fail_fast`**：当远程 Agent 不可达时，`True` 表示立即抛出异常，`False` 则跳过该 Agent 继续执行
- **`trust_remote_completion_status`**：若为 `True`，当远程 Agent 返回 completed 状态时直接采用其结果，不再让本地 LLM 判断
- **`updates`**：更新机制配置，支持 Streaming、Polling 和 PushNotification 三种模式

### 18.2.2 A2AServerConfig

`A2AServerConfig` 用于将 Crew 或 Agent 暴露为 A2A 服务：

```python
class A2AServerConfig(BaseModel):
    name: str | None = Field(default=None)
    description: str | None = Field(default=None)
    version: str = Field(default="1.0.0")
    skills: list[AgentSkill] = Field(default_factory=list)
    default_input_modes: list[str] = Field(
        default_factory=lambda: ["text/plain", "application/json"]
    )
    default_output_modes: list[str] = Field(
        default_factory=lambda: ["text/plain", "application/json"]
    )
    capabilities: AgentCapabilities = Field(
        default_factory=lambda: AgentCapabilities(
            streaming=True, push_notifications=False,
        )
    )
    protocol_version: ProtocolVersion = Field(default="0.3.0")
    server_extensions: list[ServerExtension] = Field(default_factory=list)
    transport: ServerTransportConfig = Field(default_factory=ServerTransportConfig)
    auth: ServerAuthScheme | None = Field(default=None)
```

注意 `name`、`description` 和 `skills` 字段可以留空 -- 框架会从 Crew/Agent 的定义中自动推导这些信息。AgentCard 的 `capabilities` 默认开启 streaming 但关闭 push notification。

### 18.2.3 传输协议协商

传输配置通过 `ServerTransportConfig` 和 `ClientTransportConfig` 管理，支持自动协商：

```python
class ClientTransportConfig(BaseModel):
    preferred: TransportType | None = Field(default=None)
    supported: list[TransportType] = Field(
        default_factory=lambda: cast(list[TransportType], ["JSONRPC"])
    )
    jsonrpc: JSONRPCClientConfig = Field(default_factory=JSONRPCClientConfig)
    grpc: GRPCClientConfig = Field(default_factory=GRPCClientConfig)
```

协商逻辑按优先级排列：
1. 若 client 设定了 `preferred` 且 server 支持，使用 client 偏好
2. 否则若 server 的 preferred 在 client 的 `supported` 列表中，使用 server 偏好
3. 否则在 client 的 `supported` 列表中寻找 server 支持的第一个协议

对于 gRPC 传输，`GRPCServerConfig` 提供了精细控制：

```python
class GRPCServerConfig(BaseModel):
    host: str = Field(default="localhost")
    port: int = Field(default=50051)
    tls_cert_path: str | None = Field(default=None)
    tls_key_path: str | None = Field(default=None)
    max_workers: int = Field(default=10)
    reflection_enabled: bool = Field(default=False)
```

AgentCard 签名通过 `AgentCardSigningConfig` 支持 JWS 签名，支持 RS256、ES256、PS256 等算法，确保 AgentCard 的真实性。

## 18.3 A2A Wrapper：Agent 的协议适配层

`a2a/wrapper.py` 是整个 A2A 实现的核心，约 1750 行代码。它通过 monkey-patching 的方式将 A2A 委托能力注入到 Agent 实例中。

### 18.3.1 wrap_agent_with_a2a_instance

入口函数 `wrap_agent_with_a2a_instance` 负责包装 Agent 实例：

```python
def wrap_agent_with_a2a_instance(
    agent: Agent, extension_registry: ExtensionRegistry | None = None
) -> None:
    if extension_registry is None:
        extension_registry = ExtensionRegistry()

    extension_registry.inject_all_tools(agent)

    original_execute_task = agent.execute_task.__func__
    original_aexecute_task = agent.aexecute_task.__func__

    @wraps(original_execute_task)
    def execute_task_with_a2a(self, task, context=None, tools=None):
        if not self.a2a:
            return original_execute_task(self, task, context, tools)
        a2a_agents, agent_response_model = get_a2a_agents_and_response_model(self.a2a)
        return _execute_task_with_a2a(
            self=self, a2a_agents=a2a_agents, original_fn=original_execute_task,
            task=task, agent_response_model=agent_response_model,
            context=context, tools=tools, extension_registry=extension_registry,
        )

    object.__setattr__(agent, "execute_task", MethodType(execute_task_with_a2a, agent))
    # ... 同样包装 aexecute_task, kickoff, kickoff_async
    inject_a2a_server_methods(agent)
```

设计要点：
- 使用 `object.__setattr__` 而非直接赋值，因为 Pydantic model 的 `__setattr__` 会触发验证
- 通过 `MethodType` 将函数绑定为实例方法
- 同时包装了同步和异步版本的 `execute_task` 和 `kickoff`
- 最后调用 `inject_a2a_server_methods` 注入服务端方法

### 18.3.2 委托执行流程

`_execute_task_with_a2a` 展示了完整的委托决策流程：

```python
def _execute_task_with_a2a(self, a2a_agents, original_fn, task, ...):
    original_description = task.description

    # 1. 并发获取所有远程 Agent 的 AgentCard
    agent_cards, failed_agents = _fetch_agent_cards_concurrently(a2a_agents)

    # 2. 若所有 Agent 不可达，添加通知后回退到本地执行
    if not agent_cards and a2a_agents and failed_agents:
        task.description = f"{original_description}{notice}"
        return original_fn(self, task, context, tools)

    # 3. 在 prompt 中注入远程 Agent 信息
    task.description, _, extension_states = _augment_prompt_with_a2a(
        a2a_agents=a2a_agents, task_description=original_description,
        agent_cards=agent_cards, ...
    )
    task.response_model = agent_response_model

    # 4. 让 LLM 决定是否委托
    raw_result = original_fn(self, task, context, tools)
    agent_response = _parse_agent_response(raw_result, agent_response_model)

    # 5. 若 LLM 选择委托，执行 A2A 通信
    if isinstance(agent_response, AgentResponseProtocol) and agent_response.is_a2a:
        return _delegate_to_a2a(self, agent_response=agent_response, ...)

    return agent_response.message
```

这个流程展示了一个精妙的设计：LLM 自主决定是否委托。框架在 prompt 中注入可用远程 Agent 的信息（包括 AgentCard 中的描述和技能），让 LLM 判断当前任务是否适合委托。LLM 通过结构化输出返回 `AgentResponseProtocol`，其中 `is_a2a` 字段表示是否选择委托。

### 18.3.3 并发获取 AgentCard

```python
def _fetch_agent_cards_concurrently(a2a_agents):
    agent_cards: dict[str, AgentCard] = {}
    failed_agents: dict[str, str] = {}

    max_workers = min(len(a2a_agents), 10)
    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = {
            executor.submit(_fetch_card_from_config, config): config
            for config in a2a_agents
        }
        for future in as_completed(futures):
            config, result = future.result()
            if isinstance(result, Exception):
                if config.fail_fast:
                    raise RuntimeError(...) from result
                failed_agents[config.endpoint] = str(result)
            else:
                agent_cards[config.endpoint] = result

    return agent_cards, failed_agents
```

使用 `ThreadPoolExecutor` 并发获取 AgentCard，最多 10 个并发。对于每个失败的 Agent，根据 `fail_fast` 配置决定是抛出异常还是静默跳过。

### 18.3.4 Prompt 增强与多轮对话

`_augment_prompt_with_a2a` 函数将远程 Agent 信息、对话历史和轮次控制注入到 task prompt 中：

```python
def _augment_prompt_with_a2a(a2a_agents, task_description, agent_cards,
                              conversation_history=None, turn_num=0,
                              max_turns=None, ...):
    # 构造可用 Agent 的描述（从 AgentCard 提取）
    agents_text = ""
    for config in a2a_agents:
        if config.endpoint in agent_cards:
            card = agent_cards[config.endpoint]
            agents_text += card.model_dump_json(
                indent=2, exclude_none=True,
                include={'description', 'url', 'skills'}
            )

    # 构造对话历史
    history_text = ""
    if conversation_history:
        for msg in conversation_history:
            history_text += msg.model_dump_json(...)

    # 轮次控制：临近上限时发出警告
    if max_turns is not None and conversation_history:
        turn_count = turn_num + 1
        if turn_count >= max_turns:
            warning = "CRITICAL: This is the FINAL turn. ..."
        elif turn_count == max_turns - 1:
            warning = "WARNING: Next turn will be the last."

    # Extension 处理
    if extension_registry:
        augmented_prompt = extension_registry.augment_prompt_with_all(
            augmented_prompt, extension_states
        )

    return augmented_prompt, disable_structured_output, extension_states
```

轮次控制机制特别值得关注：当对话接近 `max_turns` 上限时，系统会在 prompt 中注入 `CRITICAL` 或 `WARNING` 提示，引导 LLM 尽快结束对话。

### 18.3.5 DelegationContext 与 DelegationState

为了管理委托过程中的状态，代码定义了两个 NamedTuple：

```python
class DelegationContext(NamedTuple):
    a2a_agents: list[A2AConfig | A2AClientConfig]
    agent_response_model: type[BaseModel] | None
    current_request: str
    agent_id: str
    agent_config: A2AConfig | A2AClientConfig
    context_id: str | None
    task_id: str | None
    metadata: dict[str, Any] | None
    extensions: dict[str, Any] | None
    reference_task_ids: list[str]
    original_task_description: str
    max_turns: int

class DelegationState(NamedTuple):
    current_request: str
    context_id: str | None
    task_id: str | None
    reference_task_ids: list[str]
    conversation_history: list[Message]
    agent_card: AgentCard | None
    agent_card_dict: dict[str, Any] | None
    agent_name: str | None
```

`DelegationContext` 是不可变的委托配置，`DelegationState` 在每轮对话后更新，二者分离确保了状态管理的清晰性。

## 18.4 认证体系：auth 子模块

`a2a/auth/` 目录实现了完整的双向认证体系。

### 18.4.1 服务端认证

`server_schemes.py` 定义了五种服务端认证方案：

```python
class ServerAuthScheme(ABC, BaseModel):
    @abstractmethod
    async def authenticate(self, token: str) -> AuthenticatedUser:
        ...

class SimpleTokenAuth(ServerAuthScheme):      # 静态 Bearer Token
class OIDCAuth(ServerAuthScheme):              # OpenID Connect JWT
class OAuth2ServerAuth(ServerAuthScheme):      # OAuth2 (JWKS/Introspection)
class APIKeyServerAuth(ServerAuthScheme):      # API Key (header/query/cookie)
class MTLSServerAuth(ServerAuthScheme):        # 双向 TLS
```

所有方案共享统一的 `AuthenticatedUser` 返回类型：

```python
@dataclass
class AuthenticatedUser:
    token: str
    scheme: str
    claims: dict[str, Any] | None = None
```

`OIDCAuth` 使用 PyJWT 的 `PyJWKClient` 实现 JWKS 缓存和 JWT 验证：

```python
class OIDCAuth(ServerAuthScheme):
    issuer: HttpUrl
    audience: str
    jwks_cache_ttl: int = Field(default=3600, ge=60)
    clock_skew_seconds: float = Field(default=30.0, ge=0.0)
    required_claims: list[str] = Field(
        default_factory=lambda: ["exp", "iat", "iss", "aud", "sub"]
    )

    @model_validator(mode="after")
    def _init_jwk_client(self) -> Self:
        jwks_url = f"{str(self.issuer).rstrip('/')}/.well-known/jwks.json"
        self._jwk_client = PyJWKClient(jwks_url, lifespan=self.jwks_cache_ttl)
        return self
```

`OAuth2ServerAuth` 特别支持 Token Introspection（RFC 7662）作为 JWKS 的替代方案，适用于 Opaque Token 场景。

### 18.4.2 客户端认证

`client_schemes.py` 定义了对应的客户端认证方案：

```python
class ClientAuthScheme(ABC, BaseModel):
    tls: TLSConfig | None = Field(default=None)

    @abstractmethod
    async def apply_auth(self, client: httpx.AsyncClient,
                         headers: MutableMapping[str, str]) -> MutableMapping[str, str]:
        ...

class BearerTokenAuth(ClientAuthScheme):       # Bearer Token
class HTTPBasicAuth(ClientAuthScheme):          # HTTP Basic
class HTTPDigestAuth(ClientAuthScheme):         # HTTP Digest
class APIKeyAuth(ClientAuthScheme):             # API Key
class OAuth2ClientCredentials(ClientAuthScheme): # OAuth2 Client Credentials
class OAuth2AuthorizationCode(ClientAuthScheme): # OAuth2 Authorization Code
```

`OAuth2ClientCredentials` 实现了线程安全的 Token 管理，使用 `asyncio.Lock` 防止并发 Token 刷新：

```python
class OAuth2ClientCredentials(ClientAuthScheme):
    _access_token: str | None = PrivateAttr(default=None)
    _token_expires_at: float | None = PrivateAttr(default=None)
    _lock: asyncio.Lock = PrivateAttr(default_factory=asyncio.Lock)

    async def apply_auth(self, client, headers):
        if self._access_token is None or time.time() >= self._token_expires_at:
            async with self._lock:  # Double-check locking
                if self._access_token is None or time.time() >= self._token_expires_at:
                    await self._fetch_token(client)
        headers["Authorization"] = f"Bearer {self._access_token}"
        return headers
```

`TLSConfig` 支持 mTLS（双向 TLS），可同时为 httpx 和 gRPC 生成 SSL 凭证：

```python
class TLSConfig(BaseModel):
    client_cert_path: FilePath | None = None
    client_key_path: FilePath | None = None
    ca_cert_path: FilePath | None = None
    verify: bool = True

    def get_httpx_ssl_context(self) -> ssl.SSLContext | bool | str:
        ...

    def get_grpc_credentials(self) -> grpc.ChannelCredentials | None:
        ...
```

## 18.5 Extension 系统

CrewAI 的 A2A Extension 分为两层：客户端 Extension（wrapper 层 hook）和服务端 Extension（A2A 协议层扩展）。

### 18.5.1 客户端 Extension（A2AExtension Protocol）

定义在 `extensions/base.py` 中，使用 `@runtime_checkable` Protocol：

```python
@runtime_checkable
class A2AExtension(Protocol):
    def inject_tools(self, agent: Agent) -> None: ...
    def extract_state_from_history(
        self, conversation_history: Sequence[Message]
    ) -> ConversationState | None: ...
    def augment_prompt(
        self, base_prompt: str, conversation_state: ConversationState | None
    ) -> str: ...
    def process_response(
        self, agent_response: Any, conversation_state: ConversationState | None
    ) -> Any: ...
```

四个 hook 点覆盖了完整的对话生命周期：
1. **inject_tools**：在 Agent 被包装时注入自定义 tool
2. **extract_state_from_history**：从对话历史中提取 Extension 特定的状态
3. **augment_prompt**：在每轮对话前增强 prompt
4. **process_response**：在获得 LLM 响应后处理结果

`ExtensionRegistry` 管理多个 Extension 的协同工作，通过链式调用确保所有 Extension 都得到处理。

### 18.5.2 服务端 Extension（ServerExtension）

定义在 `extensions/server.py` 中，基于 A2A 协议标准的 Extension 机制：

```python
class ServerExtension(ABC):
    uri: str          # Extension URI，如 "urn:crewai:ext:sampling/v1"
    required: bool = False
    description: str | None = None

    @property
    def params(self) -> dict[str, Any] | None:
        return None

    def agent_extension(self) -> AgentExtension:
        return AgentExtension(
            uri=self.uri, required=self.required,
            description=self.description, params=self.params,
        )

    def is_active(self, context: ExtensionContext) -> bool:
        return self.uri in context.client_extensions

    @abstractmethod
    async def on_request(self, context: ExtensionContext) -> None: ...

    @abstractmethod
    async def on_response(self, context: ExtensionContext, result: Any) -> Any: ...
```

`ServerExtensionRegistry` 提供了隔离的 hook 调用机制 -- 单个 Extension 的异常不会影响其他 Extension：

```python
async def invoke_on_request(self, context: ExtensionContext) -> None:
    for extension in self._extensions:
        if extension.is_active(context):
            try:
                await extension.on_request(context)
                if context.server_context is not None:
                    context.server_context.activated_extensions.add(extension.uri)
            except Exception:
                logger.exception("Extension on_request hook failed",
                                 extra={"extension": extension.uri})
```

## 18.6 更新机制：Streaming、Polling 与 Push Notification

`a2a/types.py` 中的 `HANDLER_REGISTRY` 将三种更新策略映射到对应的 Handler：

```python
HANDLER_REGISTRY: dict[type[UpdateConfig], HandlerType] = {
    PollingConfig: PollingHandler,
    StreamingConfig: StreamingHandler,
    PushNotificationConfig: PushNotificationHandler,
}
```

默认使用 `StreamingConfig`，即 Server-Sent Events 流式接收。Polling 模式适用于不支持长连接的场景，PushNotification 模式则让远程 Agent 主动回调。

## 18.7 与 Google A2A 标准的关系

CrewAI 的 A2A 实现严格遵循 Google A2A 规范，体现在：

- **AgentCard 标准**：遵循 `/.well-known/agent-card.json` 的发现机制
- **消息格式**：使用 `a2a.types` 包中标准的 `Message`、`TaskState`、`Role` 等类型
- **传输协议**：支持规范中的 JSON-RPC、gRPC、HTTP+JSON
- **Extension 机制**：通过 `A2A-Extensions` HTTP header 声明 Extension，使用标准的 `AgentExtension` 对象
- **安全模型**：AgentCard 中的 `security_schemes` 与 `security` 字段符合规范

同时 CrewAI 在标准基础上做了实用性增强：
- 客户端 Extension（wrapper hook）是 CrewAI 特有的扩展机制
- 自动从 Crew/Agent 推导 AgentCard 信息
- LLM 自主决策是否委托的智能路由机制
- 多轮对话的轮次控制与渐进式警告

## 本章要点

- A2A 协议使 CrewAI Agent 既能作为 client 委托任务，也能作为 server 暴露服务接口
- `A2AClientConfig` 和 `A2AServerConfig` 分别管理客户端与服务端配置，支持传输协议自动协商（JSON-RPC/gRPC/HTTP+JSON）
- `wrap_agent_with_a2a_instance` 通过 monkey-patching 注入 A2A 能力，包装 `execute_task`、`kickoff` 等核心方法的同步与异步版本
- 委托决策由 LLM 自主完成：框架在 prompt 中注入远程 Agent 的 AgentCard 信息，LLM 通过 `AgentResponseProtocol.is_a2a` 字段表达委托意图
- AgentCard 并发获取使用 `ThreadPoolExecutor`，最多 10 并发，支持 `fail_fast` 和静默降级两种故障策略
- 认证体系覆盖服务端（SimpleToken/OIDC/OAuth2/APIKey/mTLS）和客户端（Bearer/Basic/Digest/APIKey/OAuth2 ClientCredentials/OAuth2 AuthCode），OAuth2 实现包含线程安全的 Token 刷新
- Extension 分为客户端 hook（inject_tools/augment_prompt/process_response）和服务端 A2A 协议扩展（on_request/on_response），均通过 Registry 管理
- 多轮对话通过 `DelegationContext`/`DelegationState` 管理状态，接近轮次上限时在 prompt 中注入渐进式警告
